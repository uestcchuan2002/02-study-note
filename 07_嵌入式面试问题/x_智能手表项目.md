# STM32智能手表项目
## 问题1：项目简介
>我做的是一个基于 STM32 和 FreeRTOS 的低功耗智能手表嵌入式软件项目。这个项目的主要目标是实现一个具备时间显示、心率检测、运动计步以及蓝牙通信功能的智能手表，同时重点优化设备的低功耗表现。在系统设计上，我基于 FreeRTOS 进行了多任务架构设计，将系统划分为传感器采集、蓝牙通信、数据处理、显示以及低功耗管理等多个任务，并通过队列和 StreamBuffer 实现任务之间的解耦通信。在数据处理方面，我使用 DMA 提高采集效率，并结合卡尔曼滤波算法对传感器数据进行平滑处理，从而提升数据的稳定性和准确性。在低功耗优化上，我实现了 FreeRTOS 的 Tickless 模式，并结合 RTC 定时唤醒机制，使系统在空闲时进入低功耗状态，整体待机功耗降低约30%。此外，我还引入了独立看门狗（IWDG）机制，对关键任务进行监控，并设计了异常恢复流程，提高系统的稳定性和可靠性。整体来说，这个项目让我对 RTOS 多任务调度、低功耗设计以及嵌入式系统架构有了比较深入的理解。
## 问题2：你这个智能手表项目，整体系统架构是怎么设计的？任务是怎么划分的？
>我这个智能手表项目整体是基于 FreeRTOS 做的分层架构设计，主要分为四层：
>第一层是硬件驱动层，主要包括 SPI、IIC、DMA、RTC 等外设驱动，用来和传感器、屏幕以及蓝牙模块进行数据交互。
>第二层是中间服务层，这一层主要负责对底层数据进行封装和处理，比如传感器数据的整理、蓝牙协议数据的解析，以及卡尔曼滤波等算法处理。
>第三层是 RTOS 任务层，我将系统功能拆分成多个独立任务，比如：
- 传感器采集任务（负责心率、加速度数据采集）
- 蓝牙通信任务（负责数据收发）
- 数据处理任务（滤波和数据融合）
- 显示任务（UI刷新）
- 低功耗管理任务（控制系统进入休眠）
- 看门狗监控任务（保证系统稳定运行）
第四层是应用层，主要是具体的业务逻辑，比如计步、心率显示等功能。
在任务之间的通信上：
>- 传感器数据我使用 Queue 进行传递
>- 蓝牙数据由于是流式数据，我使用了 StreamBuffer 来处理粘包问题
>- 一些状态同步使用 EventGroup
>在任务优先级设计上：
>- 传感器采集和蓝牙通信优先级较高，保证实时性
>- 数据处理居中
>- UI 和日志任务优先级较低
>在低功耗方面：
我使用了 FreeRTOS 的 Tickless 模式，并结合 RTC 做定时唤醒，在空闲任务时进入低功耗状态，从而降低整体功耗。
