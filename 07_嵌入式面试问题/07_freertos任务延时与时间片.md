# 一、FreeRTOS任务延时与时间片

## 第11章本质：任务延时机制

> **任务延时 = 从就绪列表移除 → 加入延时列表 → tick唤醒 → 回到就绪列表**

## 第12章本质：时间片调度

> **时间片 = 同优先级任务在每个 tick 进行轮转执行**

# 二、延时机制（第11章核心）

## 1 vTaskDelay 本质流程

* 当前任务：
  1. 从就绪列表移除
  2. 加入延时列表
  3. 触发调度（让出CPU）

本质区别：
* 裸机 delay：CPU 空转
* RTOS delay：CPU 给其他任务用

## 2 延时列表核心设计

### 排序规则（高频考点）

> 按 **绝对唤醒时间（xTimeToWake）排序**

关键结果：
* 只需要检查**链表头**
* 时间复杂度：
  * O(n)
  * O(1)

## 3 两个延时列表（重点难点）

### 原因

> 解决 **xTickCount 溢出问题**

### 机制

* 当前延时列表
* 溢出延时列表

当 tick 溢出：
* **两个列表交换**

一句话：
> 用双列表保证时间比较始终正确

## 4 延时任务唤醒流程（必考）

发生在：
```c
xTaskIncrementTick()
```

流程：
1. tick++
2. 检查延时列表头
3. 到期任务移出
4. 加入就绪列表
5. 触发调度（PendSV）

# 三、时间片调度（第12章核心）

## 1 时间片本质（高频）

> FreeRTOS 不是真正时间片

本质：
* 每个 tick 轮转一次

## 2 触发条件（必背）

必须同时满足：
* ≥2 个任务
* 同优先级
* 都在 Ready
* configUSE_TIME_SLICING = 1
* tick 正常运行

## 3 调度机制

每个 tick：
* 当前任务移到链表尾
* 下一个任务运行

## 4 关闭时间片的后果
```c
configUSE_TIME_SLICING = 0
```
结果：
* 同优先级不轮转
* 可能出现**任务饿死**

# 四、延时 vs 时间片（面试杀手题）

## 执行顺序（必须会）

在 tick 中断中：
1. 先处理延时任务
2. 决定是否抢占
3. 最后才时间片轮转

一句话：
> 延时唤醒优先于时间片调度

# 五、核心设计思想（高频理论题）

## 为什么用“链表管理任务”？

### 优势：
#### 1 多任务分类管理
* Ready / Blocked 分开

#### 2 提高调度效率
* 就绪列表 → O(1) 找最高优先级
* 延时列表 → O(1) 检查

#### 3 避免全局遍历
* 状态变量 → O(n)
* 链表 → O(1)

#### 4 支持复杂调度
* 抢占
* 时间片
* 延时唤醒
