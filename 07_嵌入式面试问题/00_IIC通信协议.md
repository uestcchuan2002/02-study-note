# IIC通信协议
## 1.常见电平协议
**5V TTL电平**
* 2.4 - 2.5V为高电平（逻辑1）
* 0 - 0.5V为低电平（逻辑0）

**RS232：** 负逻辑电平信号
* -15 - -3V为高电平（逻辑1）
* +3 - +15V为低电平（逻辑0）

**RS485:** 差分电平信号
RS485 用 A、B 两根差分线 传输
* 逻辑 1：Va - Vb ＜ -200mV
* 逻辑 0：Va - Vb ＞ +200mV

## 2.IIC协议

### 2.1 IIC为什么要加上拉电阻？提供高电平
1.  **实现线与逻辑**：IIC中多个设备共享同一条总线，上拉电阻可以将总线拉至高电平（空闲状态），任何设备都可以通过将总线拉低来发送信号。
2.  **适配开漏输出**：IIC总线的两个引脚都配置为开漏输出，开漏输出无法输出高电平，只能输出低电平，上拉电阻提供了将总线拉回高电平的能力。加入上拉电阻后可以让引脚输出高电平。
3.  **防止总线冲突**：当多个设备同时访问总线，上拉电阻配合开漏输出可以防止短路。
4.  **定义总线空闲状态**：上拉电阻确保总线在无设备通信时保持高电平，也就是空闲状态。

### 2.2 IIC为什么要使用开漏输出？机制设计，防止短路

1.  **实现线与功能**
    开漏输出允许多个设备同时共享同一条总线而不造成冲突，任何设备都可以通过拉低总线来发送信号，总线电平遵循“低电平优先”的逻辑。

2.  **电平转换**
    不同电压域的设备可以在同一总线上通信，只需使用与最高电压匹配的上拉电阻，即可实现跨电压域的信号兼容。

3.  **时钟同步**
    开漏结构可以使从设备通过拉低SCL线延长时钟周期，保证较慢的从设备与主设备同步，避免数据传输错误。

4.  **仲裁机制**
    在多主机环境中，开漏结构可以实现无损仲裁（低电平优先），当多个主机同时发送信号时，发送高电平的主机会自动检测到总线被拉低，从而放弃总线控制权。

### 2.3 IIC的地址位数是多少
IIC 地址主要分为 **7位地址格式** 和 **10位地址格式** 两种，其中 7 位地址是最常见的标准形式。

#### 2.3.1位地址格式（最常用）
- **传输形式**：地址传输时占 **1个字节（8位）**，由 **7位设备地址 + 1位读写控制位** 组成。
- **理论容量**：\(2^7 = 128\) 个地址。
- **可用数量**：
  - 0x00（通用呼叫地址）和 0x7F 为**保留地址**，不可用于普通设备。
  - 实际可用地址为 **126个**（\(2^7 - 2\)），部分资料也会表述为127个（仅排除0x00）。
- **读写位**：
  - 0：表示**写操作**（主设备向从设备发送数据）。
  - 1：表示**读操作**（主设备从从设备读取数据）。

#### 2.3.2 位地址格式（扩展模式）
- 为了满足更多设备挂载需求，IIC 还支持 **10位设备地址**。
- 传输时需要 **2个字节**：
  1.  第一个字节：前5位为固定的11110 + 2位地址位 + 读写位。
  2.  第二个字节：剩余8位地址位。
- **理论容量**：\(2^{10} = 1024\) 个地址，可大幅扩展总线设备数量。

### 2.4 IIC时序

IIC通信总共有三种核心信号：**开始信号**、**停止信号**、**应答信号**。

#### 1. 开始信号
- 条件：**SCL 为高电平**时，**SDA 从高电平变为低电平**
- 特征：出现**下降沿**，表示通信开始

#### 2. 停止信号
- 条件：**SCL 为高电平**时，**SDA 从低电平变为高电平**
- 特征：出现**上升沿**，表示通信结束

#### 3. 数据传输规则
- 读取数据时，**SCL 为高电平期间，SDA 电平必须保持稳定，SCL高电平期间，SDA数据位有效**
- 数据传输顺序：**MSB（高位优先）**，即先传输最高位，再传输低位

#### 4. 应答机制
- 每传输 **8 位数据** 后，接收方需要在第 9 个时钟周期给出应答：
  - **ACK（应答）**：SDA 拉低为**低电平**，表示成功接收
  - **NACK（非应答）**：SDA 保持为**高电平**，表示接收失败或结束传输

### 2.5 IIC的时钟同步和时钟拉伸是什么？

**时钟同步：** I2C总线上的SCL线是所有设备逻辑与的结果。当任一设备将SCL拉低，总线SCL就为低电平。只有当所有设备都释放SCL（高阻态），SCL才会变为高电平，这确保了最慢的设备也能跟上通信节奏。

**时钟拉伸：** 从设备可以通过持续拉低SCL来延长时钟周期。这样从设备可以有更多时间处理数据。主设备必须等待SCL实际变为高电平后才能继续。这是I2C协议中从设备控制通信速度的机制。

### 2.5 IIC多主机冲突的解决：**总线仲裁（Arbitration）**
当多个主机同时发送 Start 信号并尝试抢占总线时，IIC 协议通过**无损仲裁机制**来解决冲突，保证只有一个主机能完成通信，其他主机会自动退让。

---

**仲裁的核心原理**
IIC 总线是**开漏输出 + 线与逻辑**：
- 只要有一个设备拉低 SDA，总线就为低电平
- 只有所有设备都释放 SDA，总线才会回到高电平
- 仲裁在 **SCL 高电平期间** 对 SDA 电平进行采样比较，**低电平优先**

---

**冲突解决的完整流程**
1.**同时发起 Start 信号**
- 多个主机在 SCL 高电平时，同时将 SDA 拉低 → 都认为自己发送了 Start 信号，开始传输地址/数据。
- 此时总线 SDA 被拉低，所有主机都进入仲裁阶段。

2.**逐位比较地址/数据**
从地址字节的最高位（MSB）开始，每个主机在 SCL 高电平期间：
1.  **发送自己的 bit**：主机想发 `0` 就拉低 SDA，想发 `1` 就释放 SDA（由上拉电阻拉高）。
2.  **采样总线实际电平**：主机读取当前 SDA 总线上的真实电平。
3.  **比较判断**：
    - 如果自己发 `1`，但采样到总线是 `0` → 说明有其他主机发了 `0`，**当前主机输掉仲裁**，立即停止驱动 SDA/SCL，转为从机模式或空闲状态。
    - 如果自己发 `0`，采样到总线是 `0` → 继续下一位比较。
    - 如果自己发 `1`，采样到总线是 `1` → 继续下一位比较。

3.**仲裁结束**
- 逐位比较直到某一位出现差异：**发送 `0` 的主机获胜**，继续完成通信；发送 `1` 的主机失败，退出总线竞争。
- 如果所有主机发送的地址/数据完全一致，仲裁会持续到整个字节结束，最终只有一个主机能完成通信（通常是先发送的主机）。

---
关键特性：无损仲裁
- **无数据损坏**：输掉仲裁的主机不会破坏正在进行的通信，获胜主机可以继续完成传输，不会丢失数据。
- **无硬件损坏**：开漏输出 + 线与逻辑保证了不会出现 VCC 和 GND 短路的情况。
- **公平性**：仲裁基于发送的 bit 电平，不是基于主机优先级，保证了总线的公平使用。

---
**通俗举例**
主机A想发地址 `0x3A`（二进制 `0111010`），主机B想发地址 `0x38`（二进制 `0111000`）：
1.  第1位（bit7）：都发 `0` → 平局，继续。
2.  第2位（bit6）：都发 `1` → 平局，继续。
3.  第3位（bit5）：都发 `1` → 平局，继续。
4.  第4位（bit4）：都发 `1` → 平局，继续。
5.  第5位（bit3）：都发 `0` → 平局，继续。
6.  第6位（bit2）：主机A发 `1`，主机B发 `0` → 总线被拉低为 `0`。
7.  主机A采样到 `0`，和自己发的 `1` 不一致 → 输掉仲裁，停止驱动总线。
8.  主机B获胜，继续完成后续通信。

---
IIC 多主机冲突的解决核心是：
> **在 SCL 高电平期间逐位比较 SDA 电平，低电平优先，输掉仲裁的主机自动退让，保证通信无损完成。**
